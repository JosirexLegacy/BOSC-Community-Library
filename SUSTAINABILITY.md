\# Sustainability Strategy, BOSC Community Library



\## The Core Challenge



Most community-driven open-source projects don't fail because of bad ideas.

They fail because the people maintaining them burn out, move on, or never

had a clear picture of how the project sustains itself over time.



This document thinks through that problem honestly for the BOSC Community

Library what sustainability means for a project like this, which models

are realistic, and what the long-term path forward looks like.



\---



\## Two Models Worth Comparing



\### Model 1: Service-Based Sustainability (The Red Hat Approach)



Red Hat built a billion-dollar company on top of Linux, entirely free,

open-source software by selling support, customization, training,

and enterprise services around it. The software remained free. The

expertise and reliability did not.



Applied to this project, a service-based model might look like:



\- Offering curated, institution-specific versions of the library

&#x20; (e.g., a version organized around a specific university's curriculum)

\- Providing paid workshops or training sessions on how to use and

&#x20; contribute to the library

\- Consulting with education institutions on building their own resource

&#x20; repositories using this project as a foundation



\*\*The honest assessment:\*\* This model works at scale, with an established

user base and institutional relationships. For a project at this stage,

it is too early. There is no recurring revenue stream that wouldn't

distract from the actual work of building the library.



\---



\### Model 2: Foundation and Community-Supported Model



The alternative is the model used by Wikipedia, Mozilla, and countless

smaller OSS projects: a community of volunteer contributors, potentially

supported by small grants, institutional partnerships, or donations,

with no commercial revenue stream.



In this model, sustainability comes from:



\- Building a contributor community large enough that no single person's

&#x20; departure damages the project

\- Establishing institutional partnerships (universities, NGOs, government

&#x20; departments) that provide stability, visibility, and occasionally resources

\- Applying for small grants from foundations focused on open education

&#x20; or digital access in developing regions

\- Keeping the project's scope focused enough that it doesn't require

&#x20; full-time maintenance to stay healthy



\*\*The honest assessment:\*\* This model fits where this project is now and

where it realistically wants to go. It requires patience and community

investment rather than revenue generation.



\---



\## Recommended Approach: Community-Supported with Institutional Anchors



For the BOSC Community Library, the foundation/community model is the

right choice with one important addition: deliberate pursuit of

institutional anchors.



An institutional anchor is a university department, NGO, or government

body that formally endorses or uses the project. This does not require

money. It requires a relationship. A university that recommends this

library to its students, or a ministry that references it in its

educational materials, provides the project with:



\- Credibility that attracts more contributors

\- A stable user base that creates feedback and improvement pressure

\- Potential access to small grants or in-kind support

\- Visibility that outlasts any individual contributor's involvement



The goal in the next 12 months should be establishing at least two or

three of these institutional relationships not as clients, but as

genuine community partners.

\## Contributor Sustainability



Volunteer contributors are the engine of this project. Keeping that

engine running requires deliberate attention to a few things:



\*\*Lowering the barrier to first contribution.\*\* The CONTRIBUTING.md guide

exists precisely for this reason. But documentation alone is not enough

issues should be tagged clearly so new contributors know where to start

without needing to understand the full project first.



\*\*Recognizing contributions visibly.\*\* People contribute more consistently

when their work is acknowledged. This can be as simple as maintaining a

CONTRIBUTORS file or thanking contributors explicitly in PR merge messages.



\*\*Avoiding maintainer concentration.\*\* When one person makes 90% of the

decisions, the project is one departure away from abandonment. Distributing

review and merge responsibilities as the contributor base grows is essential.



\*\*Scoping realistically.\*\* The library should grow at the pace its

contributors can sustain, not at the pace a roadmap demands. Overambitious

scope leads to stale content, which is worse than limited content.



\---



\## Maintenance Challenges



Content-based projects have a specific maintenance problem that software

projects don't: content goes stale. A broken link, an outdated tool

recommendation, or a resource that no longer exists these erode trust

over time. The metadata footer in each resource file (including a "Last

Reviewed" date) is a first step toward managing this. But it requires

a periodic audit process ideally once per academic year where

maintainers verify that all links still work and all resources are still

relevant.



\---



\## Long-Term Vision



The long-term goal is a library that covers every major undergraduate

subject area taught in Ugandan and East African universities — built and

maintained by students and educators who understand the local context.



That goal does not require a large organization. It requires a consistent

community, clear standards, and enough institutional visibility to keep

attracting new contributors as older ones graduate and move on.



That is achievable. It just requires treating sustainability as a feature,

not an afterthought.
